# 第四章 系统设计与实现

## 4.1 系统的总体设计与实现

### 4.1.1 系统架构概述
基于AI的农业植保机器人监控软件系统采用基于Django框架的Web应用架构，采用经典的MVC（Model-View-Controller）设计模式。系统整体分为表示层、业务逻辑层、数据访问层和数据存储层，同时集成了AI服务层和设备通信层，实现了对多台农业植保机器人的集中监控、智能控制与数据分析。

**技术栈组成：**
- **后端框架**：Django 6.0.3，提供了完整的MVC架构支持
- **前端技术**：HTML5、CSS3、JavaScript、Bootstrap 5，实现响应式Web界面
- **数据库**：SQLite（开发环境），支持迁移到PostgreSQL（生产环境）
- **AI框架**：PyTorch、Ultralytics YOLOv8/YOLO11，用于农业目标检测
- **通信协议**：HTTP RESTful API、WebSocket（计划中），用于设备通信
- **数据可视化**：Chart.js、ECharts，用于监控数据展示
- **异步任务**：基于多线程的AI训练和推理任务处理

### 4.1.2 项目结构设计
系统采用模块化设计，将不同功能划分为独立的Django应用，便于维护和扩展：

```
agrirobot/                    # 主项目目录
├── agrirobot/               # 项目配置
│   ├── settings.py          # 项目设置
│   ├── urls.py              # 主路由配置
│   └── wsgi.py
├── users/                   # 用户管理模块
├── devices/                 # 设备管理模块
├── monitoring/              # 实时监控模块
├── control/                 # 远程控制模块
├── ai_recognition/          # AI识别模块
├── data_analysis/           # 数据分析模块
├── communication/           # 通信管理模块
├── templates/               # 模板文件
├── static/                  # 静态资源
├── media/                   # 媒体文件
└── manage.py                # Django管理脚本
```

### 4.1.3 数据库设计实现
系统采用Django ORM进行数据建模，通过models.py定义数据表结构，支持自动迁移和版本管理。数据库设计包含8个核心模块共22张表，实现了完整的数据关系映射：

1. **用户管理模块**：User（用户表）、LoginLog（登录日志表）
2. **设备管理模块**：RobotDevice（机器人设备表）、DeviceOperationLog（设备操作日志表）
3. **通信模块**：RobotConnection（机器人连接表）、MessageLog（消息日志表）
4. **控制模块**：ControlCommand（控制命令表）、ManualControl（手动控制状态表）
5. **AI识别模块**：AIModel（AI模型表）、TrainingTask（训练任务表）、InferenceRecord（推理记录表）等8张表
6. **监控模块**：SensorData（传感器数据表）、VideoStream（视频流表）、GPSLocation（GPS位置表）
7. **数据分析模块**：WorkSession（作业会话表）、StatisticalReport（统计报告表）、DashboardMetrics（大屏指标表）
8. **配置管理**：RecognitionConfig（识别配置表）

### 4.1.4 核心功能流程设计
系统实现了完整的农业植保机器人监控工作流：

1. **设备接入流程**：设备注册→连接建立→状态同步→数据采集
2. **监控与控制流程**：实时监控→数据分析→控制命令发送→执行反馈

4. **数据分析流程**：数据收集→统计计算→报告生成→可视化展示

### 4.1.5 安全性设计
系统实现了多层次的安全机制：
- **用户认证**：基于Django内置的认证系统，支持角色权限控制
- **数据验证**：表单验证、输入过滤、SQL注入防护
- **会话管理**：安全的会话存储和超时机制

### 4.1.6 可扩展性设计
系统采用模块化设计，支持功能扩展：
- **插件化架构**：各功能模块可独立开发、测试和部署

## 4.2 实时监控功能的设计与实现

### 4.2.1 功能概述
实时监控功能是系统的核心功能之一，负责实时采集、显示和分析农业植保机器人的状态数据、环境数据和位置数据，为用户提供全面的监控视图。

**主要监控内容：**
1. **设备状态监控**：在线/离线状态、电量、运行模式、故障信息
2. **环境数据监控**：温度、湿度、光照强度、土壤墒情
3. **位置轨迹监控**：GPS位置、移动轨迹、作业区域
4. **视频流监控**：实时视频画面、视频录制与回放

### 4.2.2 数据库设计
实时监控模块设计了3个核心数据表：

**SensorData（传感器数据表）：**
- `device`：外键关联RobotDevice，标识数据来源设备
- `temperature`、`humidity`、`light_intensity`、`soil_moisture`：环境参数
- `latitude`、`longitude`：实时位置坐标
- `speed`、`heading`：移动速度和航向角
- `recorded_at`：数据记录时间，自动创建时间戳

**VideoStream（视频流表）：**
- `device`：一对一关联RobotDevice，每个设备对应一个视频流配置
- `stream_url`：视频流地址（RTSP/RTMP/HTTP-FLV）
- `status`：流状态（活跃/不活跃）
- `started_at`、`last_frame_time`：流启动时间和最后帧时间

**GPSLocation（GPS位置表）：**
- `device`：外键关联RobotDevice
- `latitude`、`longitude`、`altitude`：精确位置坐标
- `accuracy`：定位精度
- `recorded_at`：位置记录时间

### 4.2.3 视图层实现
监控模块提供了5个核心视图函数：

1. **monitoring_dashboard()**：监控总览面板
   - 显示所有设备状态统计（在线、工作中、离线数量）

3. **sensor_data_api()**：传感器数据API接口
   - 提供JSON格式的传感器历史数据
   - 支持按时间范围和数量限制查询
   - 实现位置：`monitoring/views.py:37-56`

4. **gps_data_api()**：GPS数据API接口
   - 返回设备GPS位置历史记录
   - 支持地图轨迹展示

### 4.2.4 数据采集与处理机制
系统采用主动上报和被动查询相结合的数据采集方式：

**主动上报机制：**
- 机器人设备定期（如每秒）通过HTTP POST发送传感器数据
- 数据接收端点：`communication/views.py:84-109` 的 `robot_receive_data()` 函数
- 数据验证和存储到SensorData和GPSLocation表

**被动查询机制：**
- 前端通过AJAX定期调用`sensor_data_api()`和`gps_data_api()`
- 查询频率可配置（通常1-5秒）
- 数据库查询优化：通过`device`和`recorded_at`字段建立索引

**数据缓存策略：**
- 最新数据缓存：在Redis或内存中缓存最新传感器数据
- 历史数据分页：大数据量查询采用分页机制
- 数据聚合：定期聚合历史数据，减少存储压力

### 4.2.5 实时数据推送实现
系统采用两种方式实现数据实时更新：

1. **短轮询（Short Polling）：**
   - 前端JavaScript定时（如3秒）调用API接口
   - 后端返回最新数据，前端更新界面
   - 实现简单，兼容性好

2. **WebSocket实时推送（计划中）：**
   - 建立持久的WebSocket连接
   - 后端有新数据时主动推送到前端
   - 减少网络开销，实现真正实时

### 4.2.6 视频流处理实现
视频监控采用以下技术方案：

**视频流协议：**
- 支持RTSP、RTMP、HTTP-FLV等多种流媒体协议
- 通过VideoStream表配置每个设备的流地址
- 使用HTML5 video标签或第三方播放器（如flv.js）播放

**视频处理功能：**
- 实时播放：低延迟的视频流播放

### 4.3.4 设备心跳机制
系统通过心跳机制判断设备在线状态：

**心跳检测原理：**
- 设备定期（如每30秒）发送心跳包到服务器
- 服务器更新设备的`last_heartbeat`字段
- 前端通过计算最后心跳时间判断设备状态

**离线判断逻辑：**
```python
def check_device_online(device):
    if device.last_heartbeat is None:
        return False
    time_diff = timezone.now() - device.last_heartbeat
    return time_diff.total_seconds() < HEARTBEAT_TIMEOUT  # 默认60秒
```

**状态自动更新：**
- 定时任务（Celery）定期检查所有设备心跳
- 自动更新设备状态为离线或故障
- 发送离线告警通知

### 4.3.5 设备权限管理
系统实现多层次的设备权限控制：

**基于角色的权限控制：**
- **管理员**：可以管理所有设备，进行任何操作
- **操作员**：可以操作被授权的设备，不能删除设备
- **查看者**：只能查看设备状态，不能进行任何操作

**所有权控制：**
- 设备注册时自动分配所有者
- 所有者拥有设备的完全控制权
- 所有者可以授权其他用户操作设备

**操作权限验证：**
每个操作视图都使用`@login_required`装饰器进行登录验证，并在业务逻辑中检查具体权限。

## 4.4 AI识别功能的设计与实现

### 4.4.1 功能概述
AI识别功能是系统的智能化核心，利用深度学习技术对农业植保机器人采集的图像和视频进行分析，实现作物状态识别、病虫害检测、成熟度判断等功能，为精准农业提供决策支持。

**三大识别功能：**
1. **作物与杂草识别**：区分作物和杂草，计算密度分布，指导精准除草
2. **病虫害检测**：识别病害类型（白粉病、锈病等）和虫害类型（蚜虫、甲虫等）
3. **成熟度判断**：评估作物成熟度等级，指导精准采摘

### 4.4.2 数据库设计
AI识别模块设计了8个核心数据表，构成了完整的AI识别工作流：

**AIModel（AI模型表）：**
- `name`、`model_type`：模型名称和架构类型（YOLOv8n/YOLOv8s等）
- `model_file`、`model_path`：模型文件存储路径
- `num_classes`、`class_names`：模型类别数和类别名称
- `is_active`、`accuracy`：是否默认模型和准确率

**TrainingTask（训练任务表）：**
- `name`、`model_type`：训练任务名称和基础模型
- `dataset_path`、`epochs`、`batch_size`：训练参数
- `status`、`progress`：训练状态和进度
- `trained_model`：关联训练得到的AIModel

**InferenceRecord（推理记录表）：**
- `device`、`model`：关联设备和使用的模型
- `mode`：推理模式（单张图片、视频文件、实时摄像头）
- `source_file`、`result_file`：源文件和结果文件路径
- `confidence_threshold`、`total_detections`：置信度阈值和总检测数
- `detection_details`：检测详情（JSON格式）
- `processing_time`：处理时间

**RecognitionTask（识别任务表）：**
- `device`、`task_type`：关联设备和任务类型
- `status`、`image`、`result_json`：任务状态、分析图像和结果
- `confidence`：置信度

**结果表（3个）：**
- `CropWeedDetection`：作物杂草检测结果
- `PestDiseaseDetection`：病虫害检测结果  
- `MaturityDetection`：成熟度检测结果

**RecognitionConfig（识别配置表）：**

```python
def aggregate_work_sessions(start_date, end_date, device_id=None):
    sessions = WorkSession.objects.filter(
        start_time__date__gte=start_date,
        start_time__date__lte=end_date,
        status='completed'
    )
    
    if device_id:
        sessions = sessions.filter(device_id=device_id)
    
    total_hours = sum(s.duration for s in sessions) / 3600
    total_area = sum(s.area_covered for s in sessions)
    total_pesticide = sum(s.pesticide_used for s in sessions)
    
    return {
        'total_work_hours': total_hours,
        'total_area': total_area,
        'total_pesticide': total_pesticide,
        'session_count': len(sessions)
    }
```

## 4.7 通信管理功能的设计与实现

### 4.7.1 功能概述
通信管理功能负责系统与农业植保机器人之间的数据通信，包括连接管理、消息传输、协议解析、数据同步等，确保系统能够稳定、高效地与设备进行数据交换。

**核心通信功能：**
1. **连接管理**：设备连接建立、维护、断开和重连
2. **消息传输**：命令发送、数据接收、消息确认
3. **协议解析**：自定义通信协议解析和封装
4. **数据同步**：设备状态同步、配置同步、数据对账

### 4.7.2 数据库设计
通信管理模块设计了2个核心数据表：

**RobotConnection（机器人连接表）：**
- `device`：一对一关联RobotDevice，每个设备一个连接记录
- `ip_address`、`port`：机器人网络地址和端口
- `is_connected`：当前连接状态
- `connected_at`、`disconnected_at`：连接和断开时间
- `last_message_at`：最后消息时间，用于判断连接活跃度
- `message_count`：消息计数，统计通信量

**MessageLog（消息日志表）：**
- `connection`：外键关联RobotConnection
- `direction`：消息方向（接收inbound/发送outbound）
- `message_type`：消息类型（心跳、状态、控制命令、传感器数据等）
- `content`：消息内容，文本或JSON格式
- `timestamp`：消息时间戳

### 4.7.3 通信协议设计
系统设计了轻量级的JSON通信协议：

**消息格式：**
```json
{
  "msg_id": "unique_message_id",  // 消息唯一标识
  "timestamp": "2026-04-21T10:30:00Z",  // 时间戳
  "device_id": "robot_001",  // 设备ID
  "msg_type": "heartbeat",  // 消息类型
  "data": {  // 消息数据
    "battery": 85,
    "status": "working",
    "location": {"lat": 31.2304, "lng": 121.4737}
  },
  "checksum": "md5_checksum"  // 校验和
}
```

**消息类型定义：**
1. **heartbeat**：心跳消息，设备定期发送
2. **status_update**：状态更新，设备状态变化时发送
3. **sensor_data**：传感器数据，定期发送环境数据
4. **location_update**：位置更新，GPS位置变化时发送
5. **command_ack**：命令确认，设备收到命令后回复
6. **error_report**：错误报告，设备故障时发送
7. **config_request**：配置请求，设备请求最新配置
8. **config_ack**：配置确认，设备确认收到配置

**协议特性：**
- **轻量级**：JSON格式，易于解析和调试
- **可扩展**：消息类型可扩展，支持新功能
- **可靠**：消息ID和校验和确保消息完整性
- **有序**：支持消息顺序保证（可选）

### 4.7.4 连接管理实现
连接管理采用状态机模式：

**连接状态机：**
1. **DISCONNECTED**：未连接状态
   - 等待设备连接请求
   - 定期尝试主动连接（可选）

2. **CONNECTING**：连接中状态
   - 发送连接请求
   - 等待连接响应
   - 超时处理

3. **CONNECTED**：已连接状态
   - 正常通信
   - 心跳检测保持连接
   - 消息收发处理

4. **DISCONNECTING**：断开中状态
   - 发送断开请求
   - 清理资源
   - 状态转换到DISCONNECTED

**心跳检测机制：**
```python
def check_heartbeat(connection):
    if not connection.last_message_at:
        return False
    
    time_diff = timezone.now() - connection.last_message_at
    if time_diff.total_seconds() > HEARTBEAT_TIMEOUT:
        connection.is_connected = False
        connection.disconnected_at = timezone.now()
        connection.save()
        return False
    
    return True
```

**自动重连机制：**
- 检测到连接断开后自动尝试重连
- 重连间隔指数退避（1s, 2s, 4s, 8s, ...最大60s）
- 最大重试次数限制（如10次）
- 重连失败后发送告警通知

### 4.7.5 消息处理流程
消息处理采用生产者-消费者模式：

**消息接收流程：**
1. 设备发送消息到服务器端点 `POST /communication/robot/receive/`
2. 验证设备身份和消息完整性
3. 解析消息类型和内容
4. 记录到MessageLog表（direction='inbound'）
5. 更新RobotConnection的last_message_at和message_count
6. 根据消息类型分发到不同的处理函数
7. 返回响应给设备

**消息发送流程：**
1. 系统生成控制命令或配置消息
2. 创建MessageLog记录（direction='outbound'）
3. 通过HTTP POST或WebSocket发送到设备
4. 等待设备确认（可选，根据消息重要性）
5. 更新消息状态（已发送、已确认、超时、失败）

**消息队列管理：**
- 重要消息优先发送
- 失败消息重试机制
- 消息过期清理
- 消息去重处理
